# Contributing to Waterwall

Waterwall is a single-operator homelab tool maintained in the open. Bug reports, ideas, and
pull requests are welcome — with the understanding that the project's scope stays deliberately
narrow (see `docs/threat-model.md`).

## Reporting bugs and requesting features

Open an issue: <https://github.com/jimstratus/waterwall/issues>. The repo provides issue
templates for bugs and feature requests.

- **Bug:** include your OS/Python version, the exact command, expected vs. actual behavior, and
  relevant `journalctl -u waterwall-proxy` output.
- **Feature:** describe the problem first, then your proposed solution and any alternatives.

> **Never paste real secrets** in issues or logs. Use `AKIAIOSFODNN7EXAMPLE` or obvious fakes.

## Development setup

```bash
git clone https://github.com/jimstratus/waterwall.git
cd waterwall
python3 -m venv .venv
.venv/bin/pip install -e ".[dev]"
.venv/bin/python -m pytest          # the suite should pass: 0 failed
```

Tests do not shell out to `openssl` — they build fixtures with the `cryptography` library, so
they run portably on Linux and Windows.

## Working conventions

Bite-sized, test-driven development:

1. Write a failing test.
2. Run it; confirm it fails for the right reason.
3. Write the minimal code to pass.
4. Run it; confirm it passes.
5. Commit — **one logical change per commit** with a clear `feat(scope): …` / `fix(scope): …` /
   `test(scope): …` message (reference the spec section in the body where relevant).

- Branch from the default branch with a descriptive name (`feat/…`, `fix/…`, `docs/…`).
- Keep PRs focused; large unrelated refactors slow review.
- `ruff` and `mypy` are in the `[dev]` extra — match the surrounding style.
- New credential patterns must be tested against a synthetic-secret corpus, never real keys.

## The documentation site

The KB site under `site/` is generated by a small Python script:

```bash
.venv/bin/pip install -e ".[docs]"
python site/build.py            # output in site/_build/
```

Content lives in `site/content/*.md` and the nav in `site/nav.yaml`. Published content must not
contain real host names, IPs, or infrastructure identifiers — a test enforces this
(`tests/test_kb_build.py::test_no_internal_doc_or_host_leakage`).

## Pull requests

Open a PR against the default branch, describe what changed and why, link any related issue, and
make sure `pytest` passes. Smaller PRs land faster.

## Scope

Waterwall intentionally does not aim to be a multi-tenant service, a general DLP product, or
tamper-*proof* against a root attacker. Proposals broadening the threat model beyond a single
trusted operator are likely out of scope — but an issue is a fine place to discuss.
